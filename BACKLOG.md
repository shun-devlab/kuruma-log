## 🧭 実行ルール（運用方針）

### **基本方針**

```
1. backlog の先頭から次タスクを選ぶ
2. 実装する
3. 動作確認する
4. GitHub に反映する
5. 終わったら次のタスクへ進む
6. 重要判断が必要なときだけオーナー（しゅんさん）に確認する
```

### **判断待ちの扱い**

```
- 重要判断が必要なタスクは "blocked" として保留
- 回答待ちの間は、依存していない別タスクを進める
- 全体の開発は止めない
```

### **クローが自走してよいもの**

```
- 軽微な実装
- バグ修正
- リファクタ
- 文言調整
- UI 微修正
- backlog / docs 更新
- テスト
```

### **クローが確認を取るもの**

```
- 仕様分岐が大きいもの
- UX 判断が大きいもの
- 破壊的変更
- 外部公開や権限が絡むもの
- 優先順位変更
```

### **ステータス定義**

```
- 未開始
- 進行中
- 完了
- blocked（判断待ち / 依存待ち）
```

### **タスク完了条件（Definition of Done）**

```
- 実装した
- 動作確認した
- GitHub に反映した（または反映不能理由を明記した）
- STATUS.md を更新した
- `kuruma-log_log` に進捗報告した
```


### **外部アクセス方針（GitHub / Web / 検索）**

```
- GitHub への push / pull / API 呼び出しは必要最小限にする
- commit は小さくしすぎず、適切にまとめて push する
- 同じ操作の短時間連打を避ける
- エラー時は即リトライ連打せず、原因確認を優先する
- Web検索や取得は 1件ずつ丁寧に行い、短時間の連続アクセスを避ける
- 取得済み情報は再利用し、不要な再アクセスをしない
- 人間が普通に使う頻度に近いアクセスを基準とする
```
